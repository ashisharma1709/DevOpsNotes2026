******************** AWS Account Type ***************************


🔹 1. AWS Free Tier kya hai?

AWS Free Tier =👉 AWS ka free usage plan for beginners

✔️ Types of Free Tier
6 months free (EC2, S3 etc.)
Always free (kuch limited services)
Trial based (short duration)
✔️ Example:
EC2 → 750 hours/month free
S3 → limited storage free

⚠️ Important:
Limit cross kiya → charges start 💸
Resources band nahi kiye → bill aayega




🔑 2. Root User (Sabse powerful account)

Jab tum Amazon Web Services pe account banate ho, jo email + password use karte ho — wahi Root User hota hai.

👉 Key points:
Full access (ALL permissions)
Billing, account settings, delete account — sab kuch kar sakta hai
Isko koi restrict nahi kar sakta

⚠️ Problem:
Agar root credentials leak ho gaye → pura AWS account hack 😬
Isliye daily use ke liye NEVER use

✅ Best Practice:
Sirf initial setup ke liye use karo
MFA (Multi-Factor Authentication) enable karo
Uske baad root ko almost “lock” kar do


👤 3. IAM User (Actual working user)

IAM ka full form hai:
👉 Identity and Access Management

Ye AWS ka security system hai jahan tum users create karte ho.

👉 IAM User kya hai?

Ek individual user (developer, admin, tester, etc.)
Isko tum specific permissions dete ho

Example:
Developer → EC2 + S3 access
Tester → read-only access

👉 Key features:
Limited permissions (as per roles/policies)
Secure login (username + password / access keys)
Alag-alag users bana sakte ho team ke liye

✅ Real use:
👉 Tum khud bhi daily AWS use karne ke liye IAM user hi use karoge, root nahi



🤖 3. Service Account (AWS mein kya hota hai?)

AWS mein officially “service account” term itna direct use nahi hota (jaise GCP mein hota hai), lekin concept same hai.

👉 AWS mein iska equivalent: 👉 IAM Roles

🔐 IAM Role (Service Account jaisa)

IAM Role ek aisa identity hai jo:
Kisi user ko assign ho sakta hai Ya AWS service (like EC2, Lambda) ko

Example:
EC2 instance ko S3 access chahiye
Tum usko IAM Role attach kar dete ho

👉 Tab EC2 directly S3 access kar lega bina credentials ke



🔹 4. Real-world flow (important)
Tum AWS account banate ho → Root user milta hai
Root user se:IAM user create karo,usko permissions do, Aage ka sara kaam:👉 IAM user se karo (login karke)


🔹 5. Best Practice:

Root user:
MFA enable karo 🔐
rarely use karo

IAM user:
daily kaam ke liye use karo
roles & policies assign karo

🔥 Real-Life Analogy

Socho AWS ek company hai:

Root User → Company owner 👑
IAM User → Employees 👨‍💻
IAM Role → Temporary badges (jo kisi ko bhi diye ja sakte hain for specific work) 🎫



⚠️ Problem with Root User

Root user ek hi hota hai aur uske paas: Full access, No restrictions, Sab kuch delete/modify karne ki power


❌ Scenario 1: Team ka kaam
Agar tumhari team mein 5 log hain:
Kya sabko root password doge?
👉 Impossible (security risk 🚨)


❌ Scenario 2: Mistake risk
Agar kisi ne galti se:
EC2 delete kar diya
Database drop kar diya

👉 Root user mein no control, no limit


❌ Scenario 3: Tracking problem

Root se login hua:

Kaun kaam kar raha hai? ❓
👉 pata hi nahi chalega
✅ Solution: IAM User


Amazon Web Services ne IAM introduce kiya to solve this.

IAM user ka main purpose:

🔐 1. Security (Least Privilege)
Tum define karte ho:
Kis user ko kya access mile

Example:
Dev → sirf EC2
QA → sirf read

👉 Damage control possible


👥 2. Multiple Users
Har banda ka apna login
Password share nahi karna



📊 3. Tracking (Audit)
Kaun kya kar raha hai → logs mein dikhega
Accountability clear



🔁 4. Easy revoke
Agar koi company chhod de:
👉 uska IAM user disable/delete
Root password change karne ka jhanjhat nahi

🔧 5. Automation support
IAM users + roles use hote hain:

CLI, APIs, Scripts

👉 Root user = God mode (dangerous)
👉 IAM user = Controlled access (safe & practical)

🔥 Real-life example
Socho tum bank chala rahe ho:


Root user → bank ka owner vault key ke saath 🏦


IAM users → employees (cashier, manager, etc.)


👉 Kya tum sabko vault ki master key de doge?
👉 Nahi — isi liye IAM exist karta hai

🚀 Final takeaway
IAM user luxury nahi hai —
👉 ye AWS use karne ki basic necessity hai
